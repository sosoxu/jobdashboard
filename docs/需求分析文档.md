## 1. 文档概述

### 1.1 目的
本文档对「作业监控 Dashboard」的需求进行整理与分析，明确功能边界、用户场景、接口现状与差距、非功能性要求，作为后续设计与开发的依据。

### 1.2 范围
覆盖以下功能域：
- 作业总体运行状态监控（状态统计、趋势比对、用户 Top10）。
- 实时作业详情表（分页、过滤、自动/手动刷新、手风琴展开）。
- 作业控制（终止、重试；暂停/恢复预留）。
- 日志查看（list 日志、LOG 日志；AI 日志分析预留）。

不在本期范围：作业提交、调度策略配置、用户与权限管理后台、计费统计。

### 1.3 术语

| 术语 | 含义 |
|---|---|
| 作业监控服务 | 已上线、提供 RESTful 接口 `http://130.1.100.152:18080/job/call` 的服务 |
| BFF | 新增的 Go 后端服务，作为前端与作业监控服务/文件系统/AI 服务之间的聚合层 |
| JobState | 作业状态枚举，共 13 个值 |
| 工区 survey / 项目 project | 作业所属业务对象，记录于作业 attributes |
| 采样 | BFF 周期性调用作业监控服务接口并将结果持久化的动作 |

## 2. 项目背景

已有「作业监控服务」通过统一入口 `POST http://130.1.100.152:18080/job/call`、以 `method` 字段区分操作，提供部分能力：当前聚合统计（`GetCurrentJSFInfo`）、作业列表查询（`GetJob`）、终止作业（`Delete`）、重试作业（`Rerunmulti`）。

该服务**不提供**：历史时序数据、按用户维度的聚合统计、作业暂停/恢复、日志文件读取。本期 Dashboard 需在前端之上新增 Go BFF 层补齐上述能力。

### 2.1 作业状态枚举

```txt
enum JobState {
  jsFrozen,      // 0  冻结
  jsQueue,       // 1  排队
  jsScheduled,   // 2  已调度
  jsReady,       // 3  就绪
  jsReleased,    // 4  已释放
  jsActive,      // 5  运行中
  jsSuspending,  // 6  挂起中
  jsSuspended,   // 7  已挂起
  jsResuming,    // 8  恢复中
  jsCanceling,   // 9  取消中
  jsFinished,    // 10 已完成
  jsFailed,      // 11 失败
  jsCanceled,    // 12 已取消
};
```

### 2.2 状态映射约定（已确认）

- `jobStatus` 字段（字符串形式数字）按枚举下标映射到 JobState，例如 `"10"` → `jsFinished`。
- `GetCurrentJSFInfo` 聚合字段与枚举映射：

  | 接口字段 | 对应 JobState | 前端展示分组 |
  |---|---|---|
  | `active` | jsActive(5) | 运行中 |
  | `queue` | jsQueue(1) | 排队中（聚合展示时含 jsFrozen/jsScheduled/jsReady/jsReleased） |
  | `finish` | jsFinished(10) | 已完成 |
  | `failed` | jsFailed(11) | 失败 |
  | `canceled` | jsCanceled(12) | 已取消 |
  | `othercount` | 其余状态（2/3/4/6/7/8/9） | 其他 |

  > 注：`queue` 字段在接口侧仅返回 jsQueue 计数，前端"排队中"展示数字以接口 `queue` 为准；如需展示"等待类"合计，将在 BFF 侧基于 GetJob 采样补齐（见 4.1.1）。

## 3. 用户角色与场景

| 角色 | 场景 |
|---|---|
| 运维/管理员 | 查看整体作业运行大盘，发现异常状态（失败激增、排队堆积），定位问题作业并执行终止/重试，查看作业日志与 AI 诊断 |
| 项目负责人 | 关注本项目的作业进度与状态，按项目/工区过滤查看，必要时重试失败作业 |
| 普通用户 | 查看本人提交作业的执行情况、进度、日志 |

> 本期暂不实现登录鉴权与数据权限隔离（见第 9 节遗留问题），Dashboard 对所有访问者展示全量数据。

## 4. 功能需求

### 4.1 作业监控总体信息（Dashboard 大盘）

#### 4.1.1 作业状态统计与趋势比对
- **功能**：实时展示各状态作业数量（运行中、排队中、已完成、失败、已取消、其他），以及与前一个时刻比对的上升/下降比例。
- **比对时间间隔**：**1 分钟**（已确认）。
- **数据来源**：BFF 调用 `GetCurrentJSFInfo`（实时），比对值取自 BFF 1 分钟前的采样快照。
- **展示要素**：状态名、当前数量、较 1 分钟前变化数量与百分比（↑/↓ 箭头与颜色）、更新时间。

#### 4.1.2 作业趋势
- **功能**：展示已完成、运行中作业随时间的变化趋势。
- **时间范围**：24 小时、7 天、30 天（可切换）。
- **数据来源**：BFF 采样持久化的历史快照（见第 7 节）。
- **展示要素**：折线图，横轴时间，纵轴作业数量；图例区分"已完成""运行中"。
- **精度**：24h → 分钟级（1 分钟一个点）；7d → 小时级；30d → 天级。

#### 4.1.3 作业量用户 Top10
- **功能**：展示发送作业数量 Top10 的用户，显示用户名、数量、百分比占比。
- **统计口径**：以"当前调度中的作业"为准（基于 GetJob 全量采样的 userName 聚合）。统计窗口需确认（默认取最近 24h 提交的作业）；Top10 以外合并为"其他"。
- **数据来源**：BFF 周期性调用 `GetJob` 全量采样（分页拉取），按 userName 聚合持久化。
- **展示要素**：横向柱状图或排行表，用户名、作业数、百分比。

### 4.2 实时作业详情表

#### 4.2.1 列表展示
- **字段**：作业名称、状态、用户、进度、工区、项目、运行时间、提交时间、操作。
- **分页**：BFF 在内存全量作业快照上做本地过滤后分页（详见 4.2.2 与设计文档）。GetJob 上游分页仅用于 BFF 拉取全量快照，不直接暴露给前端。
- **状态展示**：状态枚举映射为中文标签与颜色标签（运行中-绿、排队中-蓝、已完成-灰、失败-红、已取消-橙、其他-默认）。
- **进度**：百分比进度条。
- **时间**：提交时间、运行时间（execTime）按可读格式展示。

#### 4.2.2 过滤
- **支持字段**（单一或组合、均支持多选）：
  - 作业状态 `jobStatus`（多选，枚举值）
  - 项目 `project`（多选）
  - 工区 `survey`（多选）
  - 用户 `userName`（多选）
  - 作业名称 `jobDesc`（单值文本，对作业名称做包含匹配，如名称包含 "input"）
- **多选下拉候选来源**：已有的作业监控服务**未提供**按项目/工区/用户/数据库枚举的 RESTful 接口，因此候选项由 BFF 从查询到的作业全量快照中提取 distinct 值，通过 `GET /api/v1/jobs/filters` 接口提供给前端。
- **下拉交互**：项目/工区/用户多选下拉支持 `filterable`（输入过滤）+ `collapse-tags`（折叠标签）+ `collapse-tags-tooltip`（折叠后 hover 查看全部）。
- **过滤实现**：BFF 在内存中的全量作业快照上做本地多值过滤（状态/用户/项目/工区/数据库为命中集合任一即匹配；`jobDesc` 为大小写不敏感的包含匹配），按提交时间倒序后分页返回。上游 GetJob 的 `projList`/`descFilter` 等过滤参数本期不依赖，统一在 BFF 侧本地过滤，避免上游参数语义不一致。
- **`projList` 字段格式**（已确认）：GetJob 请求/响应中的 `projList` 为对象数组，每个元素含 `database`/`project`/`survey`/`line` 四个字段，例如：
  ```json
  "projList": [{"database": "ndp_check", "project": "project1", "survey": "survey1", "line": "line1"}]
  ```
  BFF 从作业的该结构中提取 `database`/`project`/`survey` 用于展示与过滤候选。
- **`descFilter` 字段语义**（已确认）：对作业名称 `jobDesc` 做**包含**匹配（如 `descFilter="input"` 命中所有名称含 "input" 的作业）。BFF 本地过滤采用相同语义（大小写不敏感的 contains）。

#### 4.2.3 刷新
- **自动刷新间隔**：30 分钟、10 分钟、5 分钟、1 分钟、10 秒（可切换，含"关闭"选项）。
- **手动刷新**：提供刷新按钮，立即触发一次。
- **实现**：前端定时器轮询 BFF；刷新时保留当前过滤与分页状态。

#### 4.2.4 手风琴展开
- 点击作业行展开，展示基本信息摘要：作业编号、当前状态、所属用户、项目、提交时间、作业名称、执行进度、工区、运行耗时、摘要（总结作业运行情况）。
- **摘要**：基于作业字段拼装（如"作业 {jobDesc} 由 {userName} 于 {提交时间} 提交，已执行 {execTime}，进度 {jobProcess}%，状态 {状态}"）。

### 4.3 作业控制

| 操作 | 接口 | 状态 |
|---|---|---|
| 终止 | `Delete` | 已有 |
| 重试（失败作业重试） | `Rerunmulti` | 已有 |
| 暂停 / 恢复 | 无 | **预留 UI，接口待定** |

- **操作入口**：详情表"操作"列与展开面板内。
- **交互**：操作前二次确认（弹窗）；批量操作支持（勾选多作业后批量终止/重试）。
- **暂停/恢复**：本期预留按钮，点击提示"接口待提供"，不调用后端；待服务端提供接口后对接。
- **反馈**：操作成功/失败 Toast 提示，并自动刷新当前列表。

### 4.4 日志查看

#### 4.4.1 日志定位
- **list 日志**：`{项目主目录}/data/{project}/{survey}/list`
- **LOG 日志**：`{项目主目录}/data/{project}/{survey}/LOG`
- **项目主目录解析**：读取环境变量 `$NGP` 指向的 `configs/ndp/projects.conf`（XML 配置文件），按作业的 `project` 查找其 `maindir`，得到项目主目录。
- **`projects.conf` 结构**（已确认，文件已上传至 `docs/projects.conf`）：根元素 `<Projects>`，下含多个 `<project>` 子元素，每个 `<project>` 包含 `<name>`、`<dbname>`、`<maindir>` 等子元素。示例：
  ```xml
  <Projects>
    <project>
      <name>qqqq</name>
      <dbname>ndp</dbname>
      <maindir>D:\iEcoData\GeoEastData\DATA</maindir>
      ...
    </project>
  </Projects>
  ```
  BFF 按 `<name>` 匹配作业的 `project`，取对应 `<maindir>` 作为项目主目录。
- **路径安全**：拼接后的工区目录需做 `filepath.Clean` 规范化并校验是否仍以 `maindir` 为前缀，防止 `../` 路径逃逸。

#### 4.4.2 日志展示
- 分别查看 list 日志与 LOG 日志。
- 收集并分类展示 Info / Warn / Error 信息（按日志级别解析）。
- 支持级别过滤、关键字搜索、滚动加载（日志可能较大）。

#### 4.4.3 AI 日志分析
- **功能**：日志信息汇总、模块错误分布、诊断建议。
- **状态**：**预留接口待定**（AI 服务与模型后续确认）。BFF 设计可插拔适配层，本期可先以"规则诊断"占位（基于关键字/错误码的模块错误统计与常见建议库），待 AI 服务确认后替换。

## 5. 非功能需求

| 维度 | 要求 |
|---|---|
| 性能 | 大盘首屏 ≤ 2s；详情表分页查询 ≤ 1s；30 天趋势查询 ≤ 2s |
| 采样频率 | BFF 对 GetCurrentJSFInfo 采样间隔 1 分钟；GetJob 全量采样间隔可配（默认 1 分钟），避免对作业监控服务造成压力 |
| 可用性 | BFF 单实例部署，依赖作业监控服务可用；作业监控服务不可用时前端展示降级提示与最近一次缓存数据 |

| 可维护性 | 前后端分离，BFF 配置化（采样间隔、接口地址、日志根路径、AI 适配） |
| 安全 | 本期无鉴权；BFF 仅在内网部署，日志读取需做路径越权校验（防止 `../` 逃逸） |
| 数据保留 | 采样快照保留 30 天（满足 30 天趋势），超期清理 |

## 6. 接口现状与差距分析

### 6.1 已有接口（作业监控服务提供）
统一入口 `POST http://130.1.100.152:18080/job/call`：

| method | 用途 | 备注 |
|---|---|---|
| `GetCurrentJSFInfo` | 当前聚合统计 | 仅当前快照，无历史 |
| `GetJob` | 作业列表查询 | 支持分页与过滤 |
| `Delete` | 终止作业 | 入参 name 列表 |
| `Rerunmulti` | 重试作业 | 入参 name 列表 |

### 6.2 差距（由 BFF 补齐）

| 需求 | 差距 | BFF 方案 |
|---|---|---|
| 趋势比对（1 分钟前） | 无历史 | 定时采样 GetCurrentJSFInfo 持久化 |
| 24h/7d/30d 趋势 | 无时序数据 | 复用上述采样快照，按粒度聚合 |
| 用户 Top10 | 无按用户聚合 | 定时全量采样 GetJob，按 userName 聚合持久化 |
| 多选过滤（项目/工区/用户/数据库） | 上游无枚举接口、GetJob 过滤参数语义不确定 | BFF 维护全量作业内存快照，本地多值过滤 + distinct 候选值接口 |
| 暂停/恢复 | 无接口 | 前端预留，待服务端 |
| 日志读取 | 无文件接口 | BFF 读文件系统（projects.conf 解析路径） |
| AI 日志分析 | 无 AI 能力 | BFF 适配层，规则诊断占位，AI 待定 |

## 7. 数据需求（采样持久化）

BFF 需持久化以下数据（具体表结构见设计文档）：

1. **状态快照时序**：每分钟一条，记录 active/queue/finish/failed/canceled/othercount 及时间戳，用于趋势比对与 24h/7d/30d 趋势。
2. **作业全量快照（内存）**：每分钟一次全量采样（GetJob 分页拉取），写入 BFF 内存缓存（`cache.JobCache`），用于：
   - 用户 Top10 聚合（按 userName 聚合后存"用户作业量时序"，避免全量明细落库膨胀）；
   - 作业详情表的**多选过滤**与**候选值提取**（项目/工区/用户/数据库 distinct 值，供前端下拉）。
   全量明细仅驻内存不落库；列表/过滤/候选值均在内存快照上本地计算。
3. **保留策略**：30 天滚动清理。

> 说明：Top10 统计窗口与"当前调度中"范围需在评审时最终确认（见第 9 节）。
