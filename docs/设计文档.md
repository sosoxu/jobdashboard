### 2.2 分层职责

| 层 | 职责 |
|---|---|
| 前端 SPA | UI 展示、过滤/分页/刷新交互、图表渲染、轮询 BFF |
| Go BFF | 对前端的 REST API、采样调度与持久化、日志路径解析与文件读取、作业控制代理、AI 适配层 |
| 作业监控服务（已有） | GetCurrentJSFInfo / GetJob / Delete / Rerunmulti |
| 文件系统 | projects.conf 配置、项目主目录下的 list / LOG 日志 |
| AI 服务（预留） | 日志汇总、模块错误分布、诊断建议 |

### 2.3 部署拓扑

- 前端构建产物由 BFF 静态托管（同源部署，避免跨域）。
- BFF 单实例部署于内网，需同时可达：作业监控服务、日志文件系统（`$NGP` 所在主机或挂载点）。
- 建议部署端口：BFF `:8088`（可配）。

## 3. 技术选型

### 3.1 前端

| 类别 | 选型 | 说明 |
|---|---|---|
| 框架 | Vue 3 + TypeScript | 组合式 API |
| 构建 | Vite | 开发体验与构建速度 |

| 路由 | Vue Router | 单页多视图 |
| HTTP | Axios | 请求 BFF |
| 工具 | dayjs | 时间格式化 |

### 3.2 BFF（Go）

| 类别 | 选型 | 说明 |
|---|---|---|
| 语言 | Go 1.21+ | |
| Web 框架 | Gin | REST API |
| HTTP 客户端 | net/http 或 resty | 调用作业监控服务 |
| 调度器 | robfig/cron 或自研 ticker | 定时采样 |
| 持久化 | SQLite + mattn/go-sqlite3 | 轻量、单文件；可平滑切换 MySQL |
| XML 解析 | encoding/xml | 解析 projects.conf |
| 配置 | viper + 环境变量 | 采样间隔、上游地址、$NGP 路径等 |

> 选 SQLite 理由：单实例、写入量小（每分钟 1 条状态快照 + 1 次用户聚合）、零运维。如需多实例或更高吞吐，切换 MySQL 仅改 driver。

### 4.3 页面设计

#### 4.3.1 Dashboard 大盘
- 顶部：状态统计卡组（运行中、排队中、已完成、失败、已取消、其他），每卡显示数量、较 1 分钟前变化数与百分比、更新时间。
- 中部左：趋势折线图（24h/7d/30d 切换，图例：已完成、运行中）。
- 中部右：用户 Top10 横向柱状图（用户名、数量、百分比 tooltip）。
- 全局：自动刷新控制（与作业表独立或共享，默认 1 分钟）。

#### 4.3.2 作业详情表
- 顶部过滤栏（全部支持多选下拉，候选项来自 `GET /api/v1/jobs/filters`）：
  - 状态（多选，枚举值）
  - 项目（多选，`filterable` + `collapse-tags` + `collapse-tags-tooltip`）
  - 工区（多选，同上）
  - 用户（多选，同上）
  - 作业名称（单值文本输入，对 `jobDesc` 做包含匹配，如 "input"）
  - 查询按钮、重置按钮
- 右上：刷新间隔切换 + 手动刷新按钮 + 最后刷新时间。
- 表格列：勾选框 | 作业名称 | 状态 | 用户 | 进度 | 工区 | 项目 | 运行时间 | 提交时间 | 操作。
- 操作列：暂停/恢复（预留，置灰提示）、停止、重试（仅失败/已取消态可点）、查看日志。
- 行展开（手风琴）：作业编号、当前状态、所属用户、项目、提交时间、作业名称、执行进度、工区、运行耗时、摘要。
- 底部：分页（size 可选 10/20/50/100）。
- 批量操作栏：勾选后出现"批量终止""批量重试"。
- 列表数据由 BFF 在内存全量作业快照上本地过滤后分页返回（见 6.3）；过滤与分页不直接走上游 GetJob 参数。

#### 4.3.3 日志查看
- 顶部 tab：list 日志 / LOG 日志 / AI 分析。
- list / LOG：级别过滤（Info/Warn/Error 全部）、关键字搜索、虚拟滚动列表。
- AI 分析：日志汇总、模块错误分布饼图、诊断建议列表（本期规则诊断占位）。

### 4.5 自动刷新机制

```ts
// composables/useAutoRefresh.ts
export function useAutoRefresh(fetcher: () => Promise<void>) {
  const interval = ref<number>(60);     // 秒，0 表示关闭
  let timer: number | null = null;
  const lastUpdated = ref<Date | null>(null);

  const start = () => {
    stop();
    if (interval.value === 0) return;
    timer = window.setInterval(fetcher, interval.value * 1000);
  };
  const stop = () => { if (timer) { clearInterval(timer); timer = null; } };
  const refresh = async () => { await fetcher(); lastUpdated.value = new Date(); };

  watch(interval, start);
  onMounted(() => { refresh(); start(); });
  onUnmounted(stop);
  return { interval, lastUpdated, refresh };
}
```

可选间隔（秒）：`1800, 600, 300, 60, 10, 0(关闭)`。

### 5.3 采样调度器

#### 5.3.1 状态快照采样（jsf_sampler）
- 每 60s 调一次 `GetCurrentJSFInfo`（type=2），将返回的 active/queue/finish/failed/canceled/othercount 写入 `job_stats_snapshot` 表。
- 大盘"较 1 分钟前变化"查询：取最近两条快照 diff；若仅一条（首启），变化显示 0/-。

#### 5.3.2 作业全量采样（job_sampler）
- 每 60s 调 `Client.FetchAllJobs`（封装 GetJob 分页拉取，type=2），把全量作业写入共享的 `cache.JobCache`（供列表/过滤服务读取），同时按 userName 聚合计数写入 `user_job_stat` 快照。
- 限流：单次采样串行分页，page sleep 100ms，避免压垮上游。
- 降级：上游不可用时保留上一份缓存，列表/过滤接口返回 `cached=true` 与缓存时间戳。

#### 5.3.3 保留清理
- 每日凌晨 03:00 删除 `job_stats_snapshot` 与 `user_job_stat` 中 30 天前数据。

### 5.4 全量作业内存缓存（cache.JobCache）

为支持多选过滤（项目/工区/用户/数据库）及候选值提取，BFF 维护一份全量作业内存快照：

- `JobCache` 由 sampler 写入、service 读取，内部用 `sync.RWMutex` 保护。
- API：
  - `Set(jobs []model.JobInfo, ts int64)`：sampler 每轮采样后写入。
  - `Snapshot() ([]model.JobInfo, int64)`：返回当前快照副本与时间戳。
  - `Age() time.Duration` / `Empty() bool`：用于新鲜度判断。
- 列表/过滤服务在读取前调用 `ensureFresh`：若缓存为空或陈旧度 > 2× 采样间隔，则同步触发一次 `FetchAllJobs` 刷新；否则直接读缓存。
- 候选值（`Filters`）与列表过滤（`List`）均在缓存快照上本地计算，不再依赖上游 GetJob 的 `projList`/`descFilter` 参数。

## 6. 数据库设计（SQLite）

### 6.1 表结构

#### job_stats_snapshot（状态快照时序）
```sql
CREATE TABLE job_stats_snapshot (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          INTEGER NOT NULL,        -- 采样时间戳(秒)
  active      INTEGER NOT NULL DEFAULT 0,
  queue       INTEGER NOT NULL DEFAULT 0,
  finish      INTEGER NOT NULL DEFAULT 0,
  failed      INTEGER NOT NULL DEFAULT 0,
  canceled    INTEGER NOT NULL DEFAULT 0,
  othercount  INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_stats_ts ON job_stats_snapshot(ts);
```

#### user_job_stat（用户作业量时序聚合）
```sql
CREATE TABLE user_job_stat (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          INTEGER NOT NULL,        -- 采样时间戳(秒)
  user_name   TEXT    NOT NULL,
  job_count   INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_user_ts ON user_job_stat(ts);
CREATE INDEX idx_user_name_ts ON user_job_stat(user_name, ts);
```

> Top10 查询取最新 ts 的所有记录按 job_count desc limit 10；历史趋势取该用户时序。

#### sample_meta（采样元数据，便于查"最新快照")
```sql
CREATE TABLE sample_meta (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- 如 key='latest_snapshot_ts' 记录最新采样时间
```

### 6.2 数据量估算
- 状态快照：60/h × 24 × 30 = 43 200 条/月 → 极小。
- 用户聚合：假设 200 用户 × 43 200 = 8.64M 条/月 → SQLite 可承受；如用户量大可改为只存 Top N + 其他合计，或切换 MySQL/PostgreSQL。

## 8. 关键流程

### 8.1 状态统计 + 1 分钟趋势比对
```
前端刷新 → GET /dashboard/stats
  → BFF 实时调 GetCurrentJSFInfo 得当前值
  → BFF 查 job_stats_snapshot 最新一条(可能就是刚才采样写入的) 作为 prev
  → 计算 delta/deltaPct 返回
```
> 为减少上游压力，BFF 也可直接返回最近一次采样快照（≤60s 内有效）作为"当前值"，prev 取上一条。实时性与压力权衡：默认走采样缓存，提供 `?fresh=1` 强制实时。

### 8.2 趋势查询
```
GET /dashboard/trend?range=24h
  → 查 job_stats_snapshot where ts >= now-24h order by ts
  → 24h 直接返回分钟点；7d 按小时聚合(取每小时末值或平均)；30d 按天聚合
  → 返回 finish/active 序列
```

### 8.3 Top10
```
GET /dashboard/top-users
  → 查 user_job_stat 最新 ts 的全部记录
  → order by job_count desc limit 10
  → 其余合计为 others，pct = count/total
```

### 8.4 作业列表过滤
```
前端过滤条件 → GET /jobs?jobStatus=5,11&project=p1,p2&survey=s1&userName=u1&jobDesc=input
  → BFF 读取 cache.JobCache 全量快照 (ensureFresh: 空/陈旧则同步 FetchAllJobs)
  → 本地多值过滤: jobStatus/userName/project/survey/database (集合命中)
                  jobDesc (大小写不敏感 contains)
  → 按 commitTime 倒序 → 本地分页
  → 平铺 database/project/survey，生成 summary
  → 返回 {total, list, cached:true, cacheTs}

GET /jobs/filters
  → 同样读 JobCache → distinct project/survey/user/database (保持插入顺序)
  → 返回 {cacheTs, projects, surveys, users, databases}
```

### 8.5 作业控制
```
POST /jobs/control {action, names}
  → delete:  上游 method=Delete,  params={type:2, name:names}
  → rerun:   上游 method=Rerunmulti, params={type:2, name:names}
  → 汇总上游返回 → 前端 Toast + 刷新列表
```

### 8.6 日志读取
```
GET /jobs/:jobName/logs?type=list
  → 查 jobName 得 project/survey (前端可直传以省一次查询)
  → resolver: 解析 $NGP/configs/ndp/projects.conf → project.maindir
  → path = clean(maindir + "/data/" + project + "/" + survey + "/" + type)
  → 校验 path 以 maindir 为前缀
  → 读文件 → 正则分级 → 关键字过滤 → 截断返回
```

### 8.7 采样调度（后台）
```
scheduler (cron):
  - 每 60s: jsf_sampler  → GetCurrentJSFInfo → 写 job_stats_snapshot
  - 每 60s: job_sampler   → FetchAllJobs(分页 GetJob) → 写 cache.JobCache
                            → 按 userName 聚合 → 写 user_job_stat
  - 每天 03:00: cleanup   → 删 30 天前数据
```

## 10. 日志路径解析模块

### 10.1 输入
- 作业的 `project`（取自 `projList[].project`，如 `project1`）。
- 环境变量 `$NGP`。

### 10.2 `projects.conf` 结构（已确认，见 `docs/projects.conf`）
```xml
<Projects>
  <project>
    <name>qqqq</name>
    <dbname>ndp</dbname>
    <maxspace>0</maxspace>
    <maindir>D:\iEcoData\GeoEastData\DATA</maindir>
    <ssddir>...</ssddir>
    <swdisk>All</swdisk>
    ...
  </project>
  <project> ... </project>
</Projects>
```
- 根元素 `<Projects>`，下含多个 `<project>` 子元素。
- 每个 `<project>` 含 `<name>`、`<dbname>`、`<maindir>` 等。
- BFF 用 Go `encoding/xml` 严格 struct 解析：

```go
type projectsConf struct {
    XMLName  xml.Name       `xml:"Projects"`
    Projects []projectEntry `xml:"project"`
}
type projectEntry struct {
    Name    string `xml:"name"`
    DBName  string `xml:"dbname"`
    Maindir string `xml:"maindir"`
}
```

### 10.3 流程
1. 读取 `$NGP/configs/ndp/projects.conf`（首次读取后缓存，文件 mtime 变化时刷新）。
2. 严格 struct 解析为 `projectsConf`，构建 `map[project name] → maindir`（同名校验只保留首条）。
3. 按作业 `project` 查找 `maindir`；未找到返回明确错误。
4. 拼接工区目录：`{maindir}/data/{project}/{survey}`。
5. 日志文件：list → `{工区目录}/list`；LOG → `{工区目录}/LOG`。

### 10.4 安全
- `maindir`、`project`、`survey` 使用 `filepath.Clean` 规范化。
- 拼接后的工区目录必须 `HasPrefix(filepath.Clean(maindir))`，否则拒绝（防 `../` 逃逸）。
- 禁止符号链接逃逸（必要时 `os.Lstat` 校验）。

### 10.5 已知限制
- `projects.conf` 中 `<maindir>` 为 Windows 路径（如 `D:\iEcoData\...`）。若 BFF 部署在 Linux 而日志在 Windows 共享盘，需保证 `$NGP` 指向的目录结构与配置一致；跨平台路径分隔符由 `filepath.Clean` 统一处理。

## 12. 错误处理与降级

| 场景 | 处理 |
|---|---|
| 上游作业监控服务超时/不可用 | BFF 返回最近一次缓存快照 + `degraded:true` 标记，前端顶部提示"数据来自缓存" |

## 14. 风险与对策

| 风险 | 对策 |
|---|---|
| GetJob 全量采样数据量大（示例 count=19999）造成上游压力 | 分页串行 + sleep；采样间隔可调大；必要时只采最近窗口 |
| 多选过滤依赖全量快照内存占用 | 全量作业仅缓存必要字段；快照单份共享；必要时限制缓存大小 |
| 候选值/列表读陈旧缓存 | ensureFresh 在 2× 采样间隔阈值外同步刷新；前端展示 `cacheTs` |
| projects.conf 路径为 Windows 盘符，BFF 部署在 Linux | 见 10.5：依赖 `$NGP` 指向与配置一致的挂载点；`filepath.Clean` 统一分隔符 |
| AI 服务未定 | 适配层 + 规则诊断占位，不影响主流程 |
| 暂停/恢复无接口 | UI 预留，BFF 预留分支，待上游 |
| SQLite 并发写 | 采样为单写者串行，无冲突；读并发无忧 |
| 30 天数据膨胀 | 定时清理；用户量大时切换 MySQL 或仅存 Top N |
